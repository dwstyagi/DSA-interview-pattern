# Pattern 11a — Tree Fundamentals (Read Before DFS/BFS)

## Why This File Exists

Patterns 11 (BFS) and 12 (DFS) both use trees as their primary playground.
If tree recursion feels uncomfortable, every graph problem feels harder than it is.

This file covers the **building blocks** — the 6-7 tree ideas that appear
as sub-problems inside almost every medium/hard tree question.

Master these first, then go to 12_dfs.md and 11_bfs.md.

---

## The Mental Model: A Tree IS a Recursive Structure

```
A tree is either:
  - null (base case)
  - a node with a LEFT subtree and a RIGHT subtree (both are also trees)

Every tree function follows this shape:
  def fn(node)
    return BASE if node is null

    left_result  = fn(node.left)   # trust recursion to solve left subtree
    right_result = fn(node.right)  # trust recursion to solve right subtree

    combine(left_result, right_result, node.val)
  end
```

If you can write this skeleton without thinking, tree problems become fill-in-the-blank.

## Core Building Blocks

### 1. Height / Depth of a Tree

```ruby
def height(node)
  return 0 if node.nil?
  1 + [height(node.left), height(node.right)].max
end
```

**This is the single most reused pattern.** Diameter, balanced check,
max path sum — all build on top of this.

---

### 2. Count Nodes

```ruby
def count(node)
  return 0 if node.nil?
  1 + count(node.left) + count(node.right)
end
```

---

### 3. Tree Traversals (Pre / In / Post order)

```ruby
def preorder(node, result = [])    # node → left → right  (copy tree, serialize)
  return result if node.nil?
  result << node.val
  preorder(node.left, result)
  preorder(node.right, result)
  result
end

def inorder(node, result = [])     # left → node → right  (BST → sorted array)
  return result if node.nil?
  inorder(node.left, result)
  result << node.val
  inorder(node.right, result)
  result
end

def postorder(node, result = [])   # left → right → node  (delete tree, combine subtrees)
  return result if node.nil?
  postorder(node.left, result)
  postorder(node.right, result)
  result << node.val
  result
end
```

**The traversal order determines WHAT you know when you process a node:**

- Pre-order: you know the parent before children → good for copying / serializing
- In-order: you know left subtree is done → good for BST (sorted order)
- Post-order: you know BOTH subtrees are done → good for combining results up

---

### 4. Level Order (BFS on Tree)

```ruby
def level_order(root)
  return [] unless root
  q = [root]
  result = []

  until q.empty?
    level = []
    q.size.times do           # snapshot size = nodes at THIS level only
      node = q.shift
      level << node.val
      q << node.left  if node.left
      q << node.right if node.right
    end
    result << level
  end

  result
end
```

---

### 5. Check if Two Trees are Identical

```ruby
def same_tree?(p, q)
  return true  if p.nil? && q.nil?
  return false if p.nil? || q.nil?
  p.val == q.val &&
    same_tree?(p.left, q.left) &&
    same_tree?(p.right, q.right)
end
```

The structure of this function is the template for **any problem that
compares two trees in parallel** (symmetric tree, subtree check, etc.)

---

### 6. Path from Root to Node

```ruby
def find_path(node, target, path = [])
  return false if node.nil?

  path << node.val

  return true if node.val == target
  return true if find_path(node.left, target, path)
  return true if find_path(node.right, target, path)

  path.pop      # backtrack
  false
end
```

---

### 7. BST Properties

```ruby
# In a BST: inorder traversal gives sorted output
# For any node: all left descendants < node.val < all right descendants

# Search in BST
def search_bst(node, val)
  return nil if node.nil?
  return node if node.val == val
  val < node.val ? search_bst(node.left, val) : search_bst(node.right, val)
end

# Insert in BST
def insert_bst(node, val)
  return TreeNode.new(val) if node.nil?
  if val < node.val
    node.left = insert_bst(node.left, val)
  else
    node.right = insert_bst(node.right, val)
  end
  node
end
```

## Key Tricks to Remember

**Height is the #1 building block**

```ruby
# diameter    = max(left_height + right_height)   over all nodes
# balanced    = abs(left_height - right_height) <= 1  at every node
# max path sum= max(left_gain + right_gain + val) over all nodes
# all three use the same post-order skeleton, just different combination
```

**When to use pre / in / post order**

```
pre-order:  you need parent context before processing children
            → copy tree, serialize, build from traversal
in-order:   you need LEFT done first
            → BST sorted output, validate BST, kth smallest
post-order: you need BOTH children's results before processing self
            → height, diameter, max path sum, balanced check, delete tree
```

**BST shortcut — skip entire branches**

```ruby
# if target < node.val → target cannot be in right subtree → skip right
# if target > node.val → target cannot be in left subtree → skip left
# this gives O(log n) instead of O(n) for search/insert/delete
```

**Level order — always snapshot queue size**

```ruby
q.size.times { ... }    # processes exactly THIS level, not next level's nodes
# forgetting this makes everything come out on one level
```

**Null checks — check both nil at once**

```ruby
return true  if p.nil? && q.nil?   # both null → equal/symmetric → true
return false if p.nil? || q.nil?   # one null, one not → mismatch → false
# then safe to access p.val, q.val
```

**Post-order "return up" — the most powerful tree pattern**

```ruby
dfs = ->(node) {
  return 0 if node.nil?           # base case
  l = dfs.call(node.left)         # get left subtree result
  r = dfs.call(node.right)        # get right subtree result
  @ans = [@ans, combine(l, r)].max  # update global answer
  return_up(l, r, node.val)        # return contribution to PARENT
}
# the key insight: what you RETURN UP ≠ what you use to UPDATE the answer
# diameter:   return 1 + max(l,r),  update with l + r
# max path:   return val + max(l,r,0), update with l + r + val
```
